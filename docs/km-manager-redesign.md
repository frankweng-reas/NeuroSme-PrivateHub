# KB Manager 重新設計 — 知識條目中心

> 討論日期：2026-06-20

---

## 核心設計原則

**知識庫像資料庫，不像檔案管理員。**

上傳文件 = CSV 匯入資料庫，匯入完畢即結束。  
之後所有操作直接對知識條目（chunks）進行，不經過文件這一層。

---

## 現有設計 vs 新設計

### 現有（文件中心）

```
KB Manager
├── 文件列表
│   ├── SOP操作手冊.pdf [15 chunks]  ← 主角
│   └── 產品FAQ.txt [8 chunks]
└── 點文件 → chunk 列表 → 編輯 chunk
```

問題：
- 用戶要記得「這個資訊在哪個文件」才能找到
- chunk 是系統內部概念，強迫用戶理解
- 沒有自然的「直接新增一筆知識」入口
- 搜尋和編輯之間要跳多層

### 新設計（知識條目中心）

```
KB Manager
├── 🔍 搜尋知識庫...
├── [+ 新增條目]  [匯入文件]  [篩選：來源 ▾]
└── 知識條目列表
    ├── Step 1：設定網路參數      [來源：SOP.pdf]  [編輯] [刪除]
    ├── Q：如何重設密碼？         [來源：FAQ.txt]  [編輯] [刪除]
    └── 產品型號 A100 規格說明    [來源：手動新增] [編輯] [刪除]
```

## 功能設計

### 主要操作

| 操作 | 說明 |
|---|---|
| 搜尋 | 直接搜尋知識條目內容，搜尋結果即可編輯 |
| 新增條目 | 手動輸入一筆知識，不需要透過文件 |
| 匯入文件 | 批次匯入（PDF / DOCX / TXT / 網頁等），匯入後條目出現在列表 |
| 編輯條目 | 點條目直接編輯內容 |
| 刪除條目 | 刪除單筆或批次刪除 |

### 移除的操作

| 移除項目 | 原因 |
|---|---|
| ~~刪除文件~~ | 文件匯入後即結束使命，沒有刪除意義 |
| ~~文件列表頁~~ | 文件退場為「來源標籤」，不是主要操作對象 |
| ~~進文件看 chunk 列表~~ | 直接在統一列表操作，不需兩層 |

### 「想刪除一整批」的替代方案

用來源篩選 + 批次刪除，語意等同於刪除那次匯入的所有內容：

```
篩選 → 來源：SOP操作手冊.pdf
全選 → 批次刪除
```

等同於資料庫的：
```sql
DELETE FROM knowledge_entries WHERE source = 'SOP操作手冊.pdf';
```

## 後端影響評估

### 幾乎不需要改後端

現有 `km_chunks` 已是知識條目的實際儲存，只是 UI 沒有直接暴露。

| 項目 | 現況 | 需要改動 |
|---|---|---|
| 知識條目資料 | `km_chunks` | 不需改 |
| 搜尋 API | `GET /km/documents/{id}/chunks` | 需新增跨文件搜尋 `GET /km/chunks?q=xxx` |
| 編輯條目 | `PATCH /km/chunks/{id}` | 確認是否已有 |
| 刪除條目 | `DELETE /km/chunks/{id}` | 確認是否已有 |
| 批次刪除 | 無 | 需新增 `DELETE /km/chunks?document_id=xxx` |
| 文件的角色 | 主要操作對象 | 退場為來源 metadata，僅供篩選用 |

### `km_documents` 的新定位

- 保留表格，但不再是 UI 主角
- 作為「匯入紀錄」使用：誰、何時、從哪裡匯入了什麼
- 提供篩選用的 `source` 標籤
- **不再提供刪除文件的 UI 入口**
