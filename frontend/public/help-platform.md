# NeuroSme 平台使用指南

## 從工作場景選擇工具

本指南幫助你從「我想做什麼」出發，找到對應的 Agent 組合。

---

## 快速選擇表

| 我想做的事 | 推薦工具 |
|-----------|---------|
| 隨時問一個問題、腦力激盪 | **Chat** |
| 讀懂一份合約 / 報告，深入問答 | **Doc Analyst** |
| 閱讀標案 / 需求規格書，同時留下結構化記錄 | **Document Parse + Doc Analyst** |
| 從大量表單 / 名片 / 發票抽取資料 | **OCR / Vision** |
| 從 PDF 萃取固定欄位（招標文件、採購單） | **Document Parse** |
| 撰寫提案、Email、公告等文件 | **Writing** |
| 讓公司文件變成可查詢的知識庫 | **Doc Refiner → Knowledge Base** |
| 把知識庫做成 AI Bot 對外服務 | **KB Bot Builder** |
| 大量 PDF 批次轉成 Markdown 存檔 | **Doc Refiner（批次模式）** |

## 場景一：臨時問答與探索

**場景**：有個問題需要快速得到解答，或想讓 AI 幫你整理思路，腦力激盪。

### 推薦工具：**Chat**

直接提問即可，不需任何前置設定。Chat 支援：
- 附上 PDF 或圖片請 AI 解讀（4 MB 以內）
- 切換不同 LLM 模型比較回答品質
- 多輪對話保持上下文

**什麼時候該換工具？**

> 如果你是要反覆分析**同一份長文件**（例如某份合約）→ 改用 **Doc Analyst**，文件只需上傳一次，之後可繼續追問。
>
> 如果你的問題涉及**專屬知識查詢**（內部政策、產品規格）→ 改用 **Knowledge Base** 查詢，答案更準確。

---

## 場景二：深入閱讀與分析單一文件

**場景**：你拿到一份陌生的合約、技術報告或法規文件，需要快速讀懂，並針對特定條款反覆追問。

### 推薦工具：**Doc Analyst**

**流程：**
1. 上傳 PDF 或 MD 文件（可多份）
2. AI 自動生成文件摘要與建議問題
3. 點擊建議問題或自行輸入，深入追問

**適合的問法：**
- 「第五條的違約金條款如何計算？」
- 「這兩份規格書在散熱設計上有什麼差異？」
- 「請列出本報告中所有風險項目」

**注意**：Doc Analyst 專注在已上傳的文件，不查詢公司知識庫。

---

## 場景三：閱讀標案 / 需求規格書

**場景**：收到一份招標文件或需求規格書，需要同時完成兩件事：一是快速擷取關鍵欄位（截止日、預算上限、投標資格）留存記錄；二是深入讀懂技術需求、反覆追問釐清疑點。

這個場景同時需要兩種工具，分工如下：

| 工具 | 定位 | 適合的問題 |
|------|------|---------|
| **Document Parse** | 把文件變成表格 | 截止日是哪天？預算上限是多少？需要哪些資格？ |
| **Doc Analyst** | 把文件變成對話對象 | 技術規格的第三點具體要求什麼？有哪些潛在風險？ |

### 推薦流程

**方法 A：只需留存欄位記錄**

1. 開啟 **Document Parse**，選擇解析類型（如「合約解析（通用）」或「標案文件」）
2. 上傳 PDF，AI 自動填入各欄位
3. 核對後複製或另存

**方法 B：只需深入閱讀追問**

1. 開啟 **Doc Analyst**，上傳 PDF
2. AI 自動摘要並提供建議問題
3. 針對技術規格、風險條款等持續追問

**方法 C：兩者都要（最完整）**

```
Document Parse
  → 填寫關鍵欄位（截止日、預算、資格條件）
  → 點「在 Doc Analyst 分析」→ 自動跳轉，不需重新上傳
      → 繼續深入問答（技術要求、風險評估、競爭分析）
```

Document Parse 完成後有直接跳轉按鈕，文件不需重複上傳。

**適合的追問範例（Doc Analyst）：**
- 「請列出所有技術需求中，我方目前可能不符合的項目」
- 「本案的驗收標準是什麼？有哪些容易踩雷的條款？」
- 「摘要本文件的五大重點，並指出需要特別留意的段落」

## 場景四：從表單 / 圖片批量抽取資料

**場景**：有一批格式相同的文件（名片、發票、報名表、物流標籤），需要把裡面的欄位抓出來建檔。

### 推薦工具：**OCR / Vision**

**流程：**
1. 在「萃取設定」中定義欄位（例如：姓名、公司、電話）
2. 上傳圖片或 PDF，AI 自動辨識並填入欄位
3. 確認後複製或匯出 CSV

**與 Chat 的差異**：Chat 一次處理一份，無法批量；OCR / Vision 設定一次，可批量跑數百份。

---

## 場景五：從 PDF 快速萃取關鍵欄位

**場景**：手上有採購合約、工程報告或其他文字型 PDF，只需快速抓出特定欄位（工程名稱、金額、日期、負責人），不需要深入閱讀全文。

### 推薦工具：**Document Parse**

**流程：**
1. 選擇對應的解析類型（如「合約解析（通用）」）
2. 上傳 PDF，AI 逐欄填寫
3. 核對後複製；若需繼續深入問答，點「在 Doc Analyst 分析」

**與 OCR / Vision 的差異**：Document Parse 針對 PDF 文字語意做欄位萃取；OCR / Vision 以圖片辨識為主，適合名片、手寫表單、物流標籤等視覺資料。

---

## 場景六：撰寫商業文件

**場景**：需要起草一份提案、Email、會議紀要、招商說明書或產品介紹。

### 推薦工具：**Writing**

**流程：**
1. 在中欄貼入背景素材（報價明細、會議紀錄、產品規格）
2. 在指令欄說明要寫什麼（「請依以上資料撰寫一份二頁的提案書，語氣正式」）
3. AI 在右欄生成草稿，可直接在線上編輯
4. 選取段落可局部改寫（縮短、擴展、調整語氣）

**進階**：Writing 支援 Skills（指令範本）和插入模式（在游標位置 AI 生成新段落），適合長文件撰寫。

## 場景七：讓公司文件變成可查詢的知識庫

**場景**：公司有一批內部文件（HR 政策、產品手冊、SOP、客服話術），希望員工或客服 AI 能精準回答相關問題。

### 推薦工具：**Doc Refiner → Knowledge Base → KB Bot Builder**

這是一個三步驟的工作流：

```
文件整理（Doc Refiner）
        ↓
知識庫管理（Knowledge Base）
        ↓
Bot 部署（KB Bot Builder）
```

#### 步驟一：整理文件（Doc Refiner）

依文件性質選擇模式：

| 文件類型 | 使用模式 | 說明 |
|---------|---------|------|
| 問答型（FAQ、客服話術、SOP） | **文件 → FAQ** | 萃取 Q&A，適合「精準回答」Bot |
| 多章節手冊、含圖表 PDF | **文件 → 結構化 MD** | 保留章節結構，適合「AI 整合回答」Bot |
| 會議筆記、草稿 | **筆記 → FAQ** | 快速轉成一條知識 |
| 工程圖、產品圖說 | **圖片 → 結構化 MD** | 圖片轉文字說明，可附原圖 URL |
| 大量 PDF 離線轉換 | **批次 → 結構化 MD** | 一次轉多份，存到本機再上傳 |

#### 步驟二：管理知識庫（Knowledge Base）

- 建立知識庫（建議設為「公司共用」）
- 從 Doc Refiner 直接匯入，或在此上傳 `.md` / PDF
- 確認文件狀態為「就緒」後，右欄可測試查詢效果

**文件類型選擇原則：**

| 上傳類型 | 適合內容 | 說明 |
|---------|---------|------|
| 精準問答 | Q&A、SOP | 小切塊 + BM25，適合「問一句答一句」 |
| 一般文件 | 手冊、合約、報告 | 較大切塊，適合 AI 整合回答 |
| 整份查閱 | 價目表、術語表 | 不切分，整份取回 |
| 結構化 MD | Doc Refiner 產出的 .md | 依標題切塊 + BM25 |

#### 步驟三：部署 Bot（KB Bot Builder）

- 選擇知識庫 + 設定 Bot 名稱、歡迎語、主題約束
- 取得 Widget 嵌入碼或 API Key
- 可嵌入官網、企業內網或透過 API 串接現有系統

## 場景八：部署對外客服 Bot

**場景**：想在官網放一個 AI 客服，能回答產品問題、政策查詢。

### 推薦流程

1. 整理客服 FAQ → **Doc Refiner（文件 → FAQ）**
2. 匯入知識庫（公司共用） → **Knowledge Base**
3. 建立 Bot，設定主題約束（只回答產品相關問題） → **KB Bot Builder**
4. 複製 Widget 嵌入碼，貼到官網 HTML

**建議 Bot 回答模式**：FAQ 型知識庫選「精確直答」（回傳原始答案），手冊型知識庫選「AI 整合回答」（整合多段落生成回應）。

---

## 場景九：工程圖 / 產品圖說入知識庫

**場景**：製造業有大量工程圖或產品圖，圖中文字資訊（工程名稱、設備規格、施工說明）需要進入知識庫供查詢，並保留原圖連結。

### 推薦流程

1. 使用 **Doc Refiner（圖片 → 結構化 MD）**：
   - 建立萃取設定（定義要擷取哪些欄位）
   - 上傳 JPEG 或 PDF，AI 依設定萃取結構化說明
   - 點「插入 URL」加入原圖連結（格式：`[原圖連結](https://...)`）
2. 點「匯入知識庫」，選擇現有知識庫（類型自動設為 `doc_image`）
3. 查詢時 AI 會回傳文字說明，並附上原圖 URL

## Agent 功能摘要

| Agent | 主要功能 | 輸入 | 輸出 |
|-------|---------|------|------|
| **Chat** | 通用對話、隨附文件分析 | 文字 / 圖片 / PDF | 對話回應 |
| **Doc Analyst** | 深度文件問答與摘要 | PDF / MD | 對話回應 |
| **OCR / Vision** | 批量圖片欄位抽取 | 圖片 / PDF | 結構化欄位 / CSV |

## 平台管理工具

以上場景都是「使用工具」的視角。平台另有一組**管理工具**，負責維持系統正常運作，細節請參閱各自的線上說明。

> 進入方式：畫面右上角頭像 → **管理設定**（需 `admin` 角色）。

| 管理工具 | 簡介 | 角色需求 |
|---------|------|---------|
| **使用者管理** | 新增 / 停用帳號、設定角色（admin / manager / member） | `admin` |
| **使用者權限設定** | 控制各使用者可存取哪些 Agent 與功能模組 | `admin` |
| **LLM 設定** | 設定 AI 服務商（OpenAI、Gemini、Ollama 等）、可用模型清單、API 金鑰、Embedding 模型、語音模型 | `admin` |
| **LLM Skills 管理** | 建立與管理 AI 改寫指令範本，供 Writing Agent 等使用 | `admin` |
| **知識庫管理** | 跨使用者檢視與管理所有知識庫，強制重建索引、清理異常文件 | `admin` |
| **Bot 部署管理** | 集中檢視所有已部署的 Bot Widget，停用或刪除不再使用的 Bot | `admin` |
| **Agents 用量洞察** | 查看各 Agent 的使用次數、LLM token 用量與費用統計 | `admin` |

> 若 Agent 無法使用或送出按鈕為灰色，通常是尚未設定 LLM 模型或尚未取得授權，請聯絡管理員。

## 常見問題

**Q：Doc Analyst 和 Chat 都能分析文件，差別在哪？**

Chat 適合臨時、一次性的文件提問，每次對話需重新上傳。Doc Analyst 會將文件與對話綁定，下次打開可繼續追問，適合需要反覆查閱的文件。

**Q：文件已經上傳到 Knowledge Base 了，還需要 Doc Refiner 嗎？**

不一定。若文件已是結構清晰的 Markdown 或 FAQ 格式，可直接上傳到 Knowledge Base。Doc Refiner 的價值在於「把未整理的原始 PDF 自動轉成 FAQ 或結構化 MD」，節省人工整理時間。

**Q：OCR / Vision 和 Document Parse 都能從 PDF 抽資料，要選哪個？**

- 固定版式的**圖片型資料**（名片、手寫表單、物流標籤）→ **OCR / Vision**
- 文字型 **PDF**（合約、標案文件、採購單）→ **Document Parse**
- 如果不確定，先試 Document Parse；若效果不佳再換 OCR / Vision

**Q：Knowledge Base 的查詢和 KB Bot Builder 的 Bot 差別？**

Knowledge Base 的查詢是開放給有帳號的使用者做測試與維護用；KB Bot Builder 部署出的 Bot 可透過 Widget 嵌入外部網站，或透過 API 供外部系統呼叫，**不需要登入平台帳號**。
